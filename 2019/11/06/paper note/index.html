软件工程论文笔记

Posted on 2019-11-06 | Edited on 2019-11-08 | In Notes |

待整理

需要检查的数据规模：全世界。

数据更改量：一天上千次。

如果只改变一丁点地方，却对整个地图做检查，那性能开销巨大。

现在的技术依赖手工策划静态子集，运行在数据产生的不同阶段。这些硬编码子集对数据更改不了解，导致 bug 检查推迟到质量保证的下一个阶段。

不仅服务代码要用传统的基于代码的测试案例测试，数据也必须使用 data correctness checks 验证其合理性。

Apple Maps 每次发行前都要运行数十亿次数据检查，一些检查即便利用了大量的计算资源，仍然需要花费几小时。因此不可能对每次数据更改都运行全部的检查。

现在的地图团队主要在两方面让数据检查更容易处理：

  1. 把所有检查分为检查组，每次运行一组，而不是全部检查
  2. 在一批修改后运行一次检查，而不是每次修改都运行

Hand-curated 检查组采用硬编码，不考虑修改数据的类型，在手动选择的特定点执行。

我们发现这样不了解修改的硬编码会造成两个问题

  1. 检查组中的很多检查可能和修改无关
  2. 而那些有关的检查没有包含进来执行

这造成了一错误没有被发现，知道发布前的 catch-all 检查才发现。

尽可能快地找到错误。为一个新的地区绘制地图，规则可能会改变，例如。。。结果就是，为老地区开发的检查在新地区可能会不通过。

因此，我们在两方面升级了地图产品的软件工程：

  1. 动态检查推荐器，可以选择和排名，仅与最后一次修改有关的检查，帮助我们快速检测错误，节省大量的计算资源
  2. 快速渗透最后一次造成失败的特征，为整个地图开发过程提供了前所未有的可见性，以便快速通知数据编辑者和检查编辑者。

核心思路：全集能检查出的错误，子集也能够全部检查出来，我们认为它是安全的。

RQ1 Safe subset of check 与 hand-curated static set 相比，效率如何？

两个指标：

（1）effeciency：推荐子集/手工策划静态集

（2）recall：子集检查出错误/全集检查出错误

hand-curated 和 recommended sets 在每次合并请求时改变，所以一次请求，一次计算

Static analysis based check selection to recommend a safe set of checks

RQ2 Check coverage 和 safe selection 的新概念能帮助现在的手工检查选择（使用 check group）方法识别 gaps吗？

将使用候选检查的全集，计算出安全子集，确定当前手工检查操作，所遗漏的检查数量。一次请求，一次计算。

RQ3 根据失败可能性，排名检查的机器学习模型，准确性如何？

通过机器学习模型的几个标准的性能指标测量准确性

  • recall：模型预测出失败检测的百分比
  • precision：模型预测失败检测的精确度
  • FPR：误报率（不正确预测/通过的检测数量）
  • AUC：整体分类器性能度量

RQ4 机器学习结果能否提供可操作的见解，哪次编辑造成了衰退？

检验学习模型的决策部分，来认出最有识别力的信号，用它们来界定，哪一次导致了最近检查结果从通过到失败。

研究过程

地图数据由不同的团队完成，每个团队负责一个分支，最后汇总到 master 仓库。在汇总之前，需要确保他们所做的改变不会引起恶化。大量编辑和频繁的合并导致这一阶段十分关键，开销也非常大，因此我们的方法会在这一阶段进行操作。

检查静态地分成检查组，根据工程的类型由质量保障人员选择。实际的检查是从所选检查组中执行。

15500 次合并检查不通过，更改和错误原因被记录。

  1. 计算 diff（数据更改）
  2. 用 diff 来进行安全检查选择
  3. 排名安全集中的检查，使用梯度决策提升树模型和深度神经网络

RQ1：Safe Check Selection

Effeciency：消除多余检查的收益，要远大于做推荐的开销。

实验结果：推荐了 35%，65% 被消除。推荐工具开销 500 ms << 验证工作 35 s。

Recall：最大化能找到错误的检查推荐数量。理想状态，所有的失败检查，确保合并到 master 的数据没有问题。

实验结果：2.5% 能找出错误的检查没有推荐。有两个主要原因

  1. 识别对象类型和领域的元数据不准确
  2. 在分配时，仓库里已经存在的错误，不是由编辑人员引起的

RQ3: Machine Learning Based Ranking Results

使用深度神经网络和梯度提升决策树来预测检查出问题的可能性。这两个模型能够表达复杂关系，而且在很多工业应用中工作得很好。失败的可能性用来排名和选择检查。

对每一个检查，失败和通过的案例被分为 80% 训练集，20%测试集。每个检查都建立机器学习模型，pass(positive)，fail(negative)

21% 的检查使用了两种模型。性能几乎相同，但我们展现梯度提升决策树的执行结果，因为它是可解释的，并且我们可以从它的结构中获得见解。我们推测，随着训练数据规模显著增加，深度神经网络的性能会优于梯度提升决策树。

RQ4: Actionable Insights Derived from ML Models

特征重要性：输入特征对模型预测输出的重要性。

每一行代表一个物体的特征，这些字段包含物体的属性，编辑者对他们进行编辑，这些编辑用来训练模型。因此我们可以使用模型中这些字段的重要性，来理解哪次编辑造成了故障检查。

英国和加拿大的邮递区号和美国格式不同。被训练的模型显示，在编辑加拿大和英国的子邮递区号，和通过或不通过子邮递区号检查的可能性之间，有强烈的关联性。这表明那个特征对检查故障有相当大的影响，因此需要仔细检查。

数据的世界

不到一平方英里的面积，也有着巨大的信息密度，例如：单行线，双向道路，交叉，信号灯，建筑，公园，水域等等。

要构建这样的地图，我们需要遵顼特定的规则，来反映我们对世界的理解，和那些单独元素之间的关系。

对人类来说，这些都是很直白的，如果道路和水域重叠了，我们会想到那是桥或者隧道；四条单行线交叉，那这个路口不能通向所有方向；3D 地图中，建筑物不能漂浮在空中。人类可以很迅速的发现违反规则的问题。

而计算机缺乏这种直觉，这就是我们要精确编码这些规则，来检查数据的原因。

建模数据检查

我们的模型过于灵活，所有现实世界的实体都可以表示。事实上，必须加上一些额外的限制，来让模型表现理智的世界。例如，在我们的世界里，房子必须建在地上，要是你的地图里面有房子漂在空中，我们可以很自信的说那没有反应真实世界。

我们希望实体通过我们的数据正确性检查，每次检查都对现实世界中特征行为和它们彼此的关系，实施特定的规则。可表示为布尔函数 t: U -> {true, flase}. 下面是一个对交叉路口（至少有一条路通过它）的检查

建模地图修改

改变应该指出哪些检查可能会找到错误。比如我们改变了水体，那对水体的检查将会找到错误。我们为每个 Feature（row） 定义了 C，计算公式为
$$
C = {f | (f, v) ∈ (F ∆ F’)}
$$
实际应用中，通常会同时改变多个 Feature，最后得到一个改变矩阵 𝑪~i,j~ ，有改变的标签记为 1，否则为 0。

通过在改变前后的 M 中添加空行，表示整个 Feature 删除或添加。

定义数据选择安全性

检查选择算法在原始检查集中，选取一部分，这部分检查囊括了原始集中，能够暴露改版地图所有错误的检查。

因为我们要运行一套检查，所以定义检查依赖性矩阵 D~i,j~ ，表示每一条检查依赖哪些标签。
